# Пояснительная записка по проекту

## 1. Название проекта

`Drone Delivery Platform` - backend for frontend платформа для сервиса доставки еды дронами.

## 2. Выбор категории пользователей по требованиям

По требованиям проект должен относиться к одной из категорий:

- доступ неограниченного круга внешних пользователей, контрагентов компании: клиенты, поставщики, партнеры;
- доступ сотрудников компании, которые работают удаленно или с мобильных устройств в рамках предметной области.

Для этого проекта выбрана первая категория:

```text
Сервис для внешних клиентов, которые заказывают еду из магазинов и ресторанов с доставкой дроном.
```

Это основной сценарий проекта. Пользователь `client_1`, `client_2`, `client_3` представляет внешнего клиента сервиса. Клиент регистрируется/авторизуется, выбирает еду из каталога, создает заказ и отслеживает миссии доставки.

При этом в системе есть внутренние роли:

- `operator_1` - сотрудник компании, который удаленно управляет операционным контуром доставки: видит новые миссии, готовые дроны и телеметрию;
- `manager_1` - сотрудник компании, который смотрит метрики и состояние миссий.

Наличие оператора не меняет выбранную категорию проекта. Оператор нужен для обслуживания бизнес-процесса, а основная ценность сервиса направлена на внешний круг клиентов. Поэтому формулировка для защиты:

```text
Проект реализует B2C/B2B2C сервис доставки еды дронами для внешних клиентов. Внутренние роли оператора и менеджера используются для обработки заказов, мониторинга дронов и контроля исполнения доставки.
```

## 3. Предметная область

Предметная область - доставка еды из магазинов и ресторанов с использованием беспилотных летательных аппаратов.

Сервис решает задачу быстрого оформления и контроля доставки:

1. Клиент получает доступ к каталогу еды.
2. Клиент создает заказ из позиций каталога.
3. Система создает миссию доставки.
4. Оператор видит новые миссии и готовые дроны.
5. Дрон выполняет доставку.
6. Телеметрия дрона доступна backend-слою.
7. Менеджер контролирует метрики и статусы.

## 4. Почему проект соответствует выбранной категории

Проект подходит под категорию внешних пользователей, потому что:

- внешний клиент работает с сервисом через клиентский сценарий `/api/bff/customer/dashboard/`;
- клиентские заказы создаются пользователями, не являющимися сотрудниками компании;
- каталог еды предназначен для выбора товаров внешним пользователем;
- backend использует JWT, значит клиент может работать удаленно через web/mobile приложение;
- frontend prototype показывает внешний интерфейс клиента;
- оператор и менеджер являются служебными ролями компании и не подменяют основной сценарий.

## 5. Пользовательские роли

### Клиент

Клиент - внешний пользователь сервиса.

Возможности:

- авторизоваться;
- просмотреть профиль;
- просмотреть список магазинов;
- увидеть превью каталога еды;
- создать заказ;
- увидеть свои миссии доставки.

Основные endpoints:

- `POST /api/token/`
- `GET /api/bff/customer/dashboard/`
- `POST /api/bff/customer/dashboard/`

### Оператор

Оператор - внутренний сотрудник компании.

Возможности:

- увидеть новые миссии;
- увидеть доступные дроны;
- увидеть последнюю телеметрию;
- использовать операционный интерфейс для контроля доставки.

Основной endpoint:

- `GET /api/bff/operator/control/`

Доступ разрешен только пользователям из группы `Operators` или superuser.

### Менеджер

Менеджер - внутренний сотрудник компании.

Возможности:

- увидеть метрики по дронам;
- увидеть количество миссий по статусам;
- посмотреть список миссий;
- контролировать общее состояние сервиса.

Основной endpoint:

- `GET /api/bff/manager/dashboard/`

Доступ разрешен только пользователям из группы `Managers` или superuser.

## 6. Архитектурный подход

В проекте применен подход Backend for Frontend.

Это означает, что frontend не ходит напрямую во множество мелких доменных endpoints для сборки экрана. Вместо этого frontend обращается к BFF endpoints, которые уже агрегируют данные под конкретный пользовательский сценарий.

Пример:

- клиентскому экрану нужны профиль, список магазинов, каталог и миссии;
- без BFF frontend должен был бы делать несколько запросов;
- с BFF frontend делает один запрос `GET /api/bff/customer/dashboard/`;
- backend возвращает структуру, готовую для отображения.

## 7. Структура backend

```text
accounts/
  CustomUser

delivery/
  Drone
  DeliveryMission
  CatalogItem
  API для миссий, дронов и каталога

telemetry/
  Telemetry
  API для логов телеметрии и аварийной остановки

bff/
  CustomerDashboardBFFView
  ManagerDashboardBFFView
  OperatorControlBFFView
  HealthCheckView
```

## 8. Структура frontend prototype

```text
frontend-prototype/
  index.html
  styles.css
  app.js
  README.md
```

Frontend prototype - это отдельный статический клиент. Он не является Django templates и не нарушает BFF-архитектуру.

Назначение прототипа:

- показать экраны клиента, оператора и менеджера;
- проверить JWT-авторизацию;
- проверить работу BFF endpoints;
- продемонстрировать, что внешний клиент работает через HTTP API.

## 9. Основные сущности базы данных

### CustomUser

Кастомная модель пользователя.

Используется для:

- клиентов;
- операторов;
- менеджеров;
- superuser.

### CatalogItem

Позиция каталога еды.

Поля:

- `store` - магазин или ресторан;
- `name` - название еды;
- `price` - цена.

### DeliveryMission

Миссия доставки.

Поля:

- `customer` - клиент, создавший заказ;
- `drone` - назначенный дрон;
- `delivery_address` - адрес доставки;
- `order_content` - нормализованный состав заказа;
- `status` - статус миссии.

`order_content` хранит не произвольный текст, а нормализованные позиции еды:

```json
{
  "item_id": 17,
  "store": "vkusvill",
  "store_name": "ВкусВилл",
  "name": "Бургер с говядиной",
  "quantity": 2,
  "unit_price": 420.0,
  "line_total": 840.0
}
```

### Drone

Дрон доставки.

Поля:

- `serial_number`;
- `model_name`;
- `battery_level`;
- `status`.

### Telemetry

Телеметрия дрона.

Поля:

- `drone`;
- `latitude`;
- `longitude`;
- `altitude`;
- `timestamp`.
